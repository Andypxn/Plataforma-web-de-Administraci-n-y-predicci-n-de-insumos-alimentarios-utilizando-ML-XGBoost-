Fase 2: Agregación Temporal de la Demanda

Objetivo: Transformación de Granularidad Transaccional a Unidades de Tiempo Discretas (ABT)

Tesis: Optimización de Inventarios mediante XGBoost

1. Cambio de Paradigma: Del Ticket al Día

En la Fase 1 consolidamos las tablas SQL. Sin embargo, los modelos de predicción de inventario no operan sobre transacciones individuales (ventas aisladas), sino sobre el volumen total de demanda que el almacén debe satisfacer en un periodo determinado.

Qué se hizo: Se aplicó un proceso de Temporal Resampling para agrupar miles de transacciones individuales en registros diarios únicos.
Teoría de Granularidad: La demanda agregada reduce la varianza estocástica de las transacciones individuales (ruido) y permite identificar señales estacionales claras (como el aumento de ventas los días viernes).
Impacto en ML: XGBoost requiere una variable objetivo continua (Ventas_Totales_Día) para realizar una regresión supervisada eficaz.

2. Lógica de Agregación y Reducción de Dimensionalidad

Utilizamos funciones de agregado para colapsar la dimensión transaccional manteniendo la riqueza informativa:

SELECT DATE(fecha_orden) AS fecha_dim, -- Reducción de granularidad SUM(cantidad_unidades) AS demanda_total, -- Variable Objetivo (Target) COUNT(id_orden) AS flujo_transacciones, -- Feature de intensidad AVG(total_venta) AS ticket_promedio -- Feature de comportamiento FROM v_consolidacion_maestra GROUP BY DATE(fecha_orden) ORDER BY fecha_dim;

El resultado de esta fase es la ABT, la estructura de datos definitiva para el entrenamiento del modelo.

4. Integridad Avanzada: Tratamiento de Vacíos y Sesgos

Para garantizar una defensa sólida ante un tribunal técnico, se implementaron controles de rigor sobre la estructura de la ABT:

A. Imputación de Demanda Cero (Zero-Inflation Handling):

En una base de datos relacional, los días sin ventas son "invisibles". Para un modelo predictivo, la ausencia de demanda es una señal crítica.

  • Solución: Se generó una serie cronológica completa (Date Spine) mediante un LEFT JOIN.
  • Impacto: Evita el Sesgo Optimista del modelo, asegurando que XGBoost aprenda a identificar días de baja actividad o cierres.
B. Prevención de Look-Ahead Bias (Sesgo de Anticipación):

Un error común es incluir información del futuro en el vector de entrenamiento (ej. usar el ticket promedio del día T para predecir la demanda del día T).

  • Solución: Las variables de comportamiento se calculan con un desfase (Lagging), utilizando únicamente información disponible al cierre del día T-1.
C. Conservación de Masa de Datos:

Validación mediante sumatorias cruzadas:

Σ UnidadesTransaccionales = Σ UnidadesAgregadas(ABT)

Este control asegura que el proceso de agregación no incurrió en pérdida de registros ni errores de desbordamiento en tipos NUMERIC.

5. Justificación Técnica para la Tesis

La agregación temporal no es solo una simplificación; es una necesidad técnica por tres razones:

  1. Alineación Operativa: Las compras a proveedores se realizan por días o semanas, no por cada venta que entra al sistema.
  2. Estabilidad del Gradiente: XGBoost converge más rápido cuando la varianza del target es representativa de un ciclo (24h).
  3. Detección de Patrones: Permite inyectar Features Temporales (ej. ¿Es fin de mes?) que no tendrían sentido a nivel de ticket individual.
