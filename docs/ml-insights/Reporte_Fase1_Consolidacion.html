1.1. Valor Predictivo por Tabla (The Signal Mapping)

📊 Ventas (ordenes_de_ventas)

Señal: Estacionalidad y Tendencia Temporal. Proporciona el "cuándo". Permite al modelo detectar ciclos semanales, quincenales y festivos. Sin esta tabla, el modelo no tendría noción del tiempo, solo del volumen.

📦 Detalles (detalles_ordenes_de_ventas)

Señal: Granularidad de Demanda. Es la fuente de nuestra variable objetivo (Target). Proporciona el "cuánto" por cada item individual, permitiendo predicciones a nivel de SKU.

📉 Inventarios (inventarios)

Señal: Restricciones de Suministro. Aporta variables de control como el stock actual y límites críticos. Ayuda al modelo a entender si una baja venta fue por falta de demanda o por falta de producto (Out-of-stock signal).

💰 Historiales (precios/costos)

Señal: Elasticidad y Factibilidad. Los cambios en precios afectan la demanda (Elasticidad). Los costos afectan la prioridad de compra. Son vitales para optimizar "¿A qué precio comprar?".

2. Proceso de Consolidación (The Joins Deep Dive)

La consolidación es el arte de transformar un esquema normalizado en una estructura plana para Machine Learning mediante un uso riguroso de álgebra relacional.

SELECT ov.fecha_orden, p.nombre, dov.cantidad_pz, i.cantidad_pz as stock
FROM ordenes_de_ventas ov
INNER JOIN detalles_ordenes_de_ventas dov ON ov.id = dov.ordenes_de_ventas_id
INNER JOIN productos p ON dov.productos_id = p.id
LEFT JOIN inventarios i ON p.id = i.productos_id

INNER JOIN: El Filtro de Calidad

Justificación Técnica: Se utiliza INNER JOIN para la relación Venta-Detalle-Producto porque requerimos una Coincidencia de Intersección Estricta. En términos de entrenamiento de modelos, un dato de venta sin detalle o sin producto es "ruido" o "dato corrupto". Al usar INNER, garantizamos que el dataset resultante contenga únicamente registros con integridad referencial completa.

LEFT JOIN: El Seguro de Continuidad

Justificación Técnica: Se utiliza LEFT JOIN para conectar con Inventarios o Historiales. ¿Por qué? Para evitar el borrado accidental de registros. Si un producto se vendió pero, por un error operativo, no tiene un registro de inventario, un INNER JOIN eliminaría la venta del histórico. El LEFT JOIN preserva la venta y nos permite manejar el valor nulo en la capa de limpieza, evitando sesgos de sub-representación en el modelo.

3. Análisis de Tipos de Datos e Integridad Técnica

La elección de tipos de datos no es estética, es una decisión de Optimización de Cómputo e Integridad Financiera.

TIMESTAMPTZ: El Guardián del Tiempo

Análisis Técnico: PostgreSQL utiliza TIMESTAMPTZ para normalizar cualquier fecha ingresada al estándar UTC (Universal Time Coordinated) antes de escribirla en el disco. Esto garantiza la Inmutabilidad Temporal.

  • ISO 8601: Se utiliza este estándar (ej. 2026-02-20T14:00:00Z) como protocolo de comunicación universal entre Java, Postgres y Python.
  • Manejo de Zonas Horarias: Al extraer los datos, la plataforma los proyecta a la zona horaria local, pero la base siempre es UTC. Esto elimina el Timezone Drift (desfase de horas), asegurando que el análisis de estacionalidad (ej. "Viernes noche") sea real y no un error de configuración del servidor.

NUMERIC(19,2): Contra el Error IEEE 754

Análisis Técnico: A diferencia de FLOAT o REAL, que utilizan representación binaria aproximada (estándar IEEE 754), NUMERIC es un tipo de Punto Fijo Exacto.

  • Rounding Errors: Los tipos FLOAT acumulan errores infinitesimales en sumatorias masivas (ej. 0.1 + 0.2 != 0.3). En un sistema POS de tesis, esto invalidaría la **Auditoría Financiera**.
  • Precisión: Usamos (19,2) para soportar grandes volúmenes de dinero con exactitud decimal absoluta, garantizando que el modelo ML aprenda de costos reales, no de aproximaciones.

UUID / VARCHAR(36): Unicidad en la ABT

Análisis Técnico: El uso de Identificadores Únicos Universales (UUID) es superior a los enteros autoincrementales en procesos de Ingeniería de Datos.

  • Fusión de Fuentes: Al crear la Analytical Base Table (ABT), fusionamos datos de múltiples tablas. Los UUID garantizan que no existan colisiones de llaves primarias, asegurando que cada fila en el dataset de entrenamiento sea única y trazable hasta la transacción original.

4. De SQL a la Analytical Base Table (ABT)

La ABT es el paso final de la Fase 1. Es una **Desnormalización Agregada** que transforma el caos transaccional en un vector de características que XGBoost puede interpretar.

¿Qué se hizo? Agregamos por Día + Producto.

¿Por qué? XGBoost no entiende de "ventas individuales", entiende de patrones. Al aplanar los datos, creamos columnas como dia_semana, promedio_precio y stock_inicial, facilitando la detección de correlaciones no lineales.

5. Resumen Pedagógico para Sínodos

Pregunta del Sínodo Respuesta Técnica Maestra
¿Por qué normalizar fechas? Para evitar el Timezone Drift que corrompe el análisis de estacionalidad en series de tiempo.
¿Por qué usar NUMERIC en lugar de FLOAT? Para garantizar integridad financiera y evitar la propagación de errores de redondeo del estándar IEEE 754.
¿Qué es una ABT? Es la Analytical Base Table, una vista desnormalizada que representa el vector de características (Feature Vector) para el modelo.
