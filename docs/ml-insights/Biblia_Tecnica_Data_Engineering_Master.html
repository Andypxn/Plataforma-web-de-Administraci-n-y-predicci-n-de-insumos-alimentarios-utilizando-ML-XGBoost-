BIBLIA TÉCNICA: PIPELINE PREDICTIVO

TESISTA: Andres Jimenez FECHA EXAMEN: 12 MARZO 2026 MODELO: XGBoost v2.0

Este documento unifica las tres fases críticas de la ingeniería de datos necesarias para transformar el caos transaccional de un POS en un motor de predicción de inventarios de alta precisión. La coherencia entre la base relacional, la agregación temporal y la extracción de señales es lo que garantiza la estabilidad del modelo.

Fase 1: La Capa Relacional e Integridad

El fundamento del sistema. Aquí garantizamos que los datos sean trazables, exactos y sin errores de redondeo financiero.

Consolidación mediante Joins Estratégicos

Se implementó una vista maestra utilizando INNER JOIN para la intersección estricta de ventas-productos (limpieza de datos corruptos) y LEFT JOIN para inventarios (preservación de historia ante fallos de stock).

IMPACTO: Integridad Referencial del 100% en el Vector de Entrenamiento.

Tipado de Datos de Grado Financiero

  • NUMERIC(19,2): Evita el error IEEE 754 de los tipos FLOAT, asegurando que el modelo aprenda de costos reales.
  • TIMESTAMPTZ: Normalización UTC obligatoria para eliminar el Timezone Drift en el análisis de estacionalidad.

Fase 2: Agregación Temporal y Tratamiento de Sesgos

La transición de la granularidad de "tickets" a "días". Aquí es donde el modelo adquiere su capacidad de entender el tiempo.
-- Agregación Maestra por Día SELECT DATE(fecha) as fecha_id, SUM(unidades) as demanda_total FROM v_consolidada GROUP BY 1;

Solución al Problema del "Vaciado Temporal"

Se aplicó una técnica de Date Spine (Espinazo de Fechas) para imputar registros de demanda cero en días sin ventas.

DEFENSA: Elimina el "Sesgo Optimista" permitiendo que el modelo detecte días de inactividad.

Prevención de Look-Ahead Bias

Se garantiza que las métricas de comportamiento se calculen exclusivamente con datos del pasado (T-1), asegurando que el modelo no "vea el futuro" durante el entrenamiento.

Fase 3: Feature Engineering y Extracción de Señales

La transformación final. Convertimos fechas y volúmenes en señales predictivas que XGBoost puede utilizar para ramificar sus árboles de decisión.
Feature Vectort = {Ciclicidadt, Memoria(Lag)t-1,t-7, Inercia(MA7)t}
Categoría Feature Generada Justificación Técnica en XGBoost
Ciclicidad dia_semana, es_quincena Permite particiones binarias para capturar estacionalidad local.
Memoria Lag_1, Lag_7 Proporciona auto-correlación (el mejor predictor es el pasado reciente).
Suavizado Rolling_Mean_7 Reduce el ruido de picos atípicos y muestra la tendencia real.

Codificación para Árboles (Label Encoding)

Se evitó el One-Hot Encoding masivo para prevenir la "maldición de la dimensionalidad". XGBoost maneja eficientemente las variables ordinales mediante particiones por umbrales.

Fase 4: Serialización y Eficiencia de Memoria

El paso final antes del entrenamiento. La transformación del objeto de alto nivel (Pandas DataFrame) a una estructura de bajo nivel optimizada para el hardware.

Transición a NumPy: Memoria Contigua de 64 bits

Para que XGBoost pueda procesar los datos a máxima velocidad, los DataFrames de Pandas se transforman en arreglos de NumPy (ndarrays) con un diseño de memoria específico.

  • Precisión de 64 bits (float64): Se utiliza doble precisión para mantener la fidelidad de los datos financieros procesados en SQL, minimizando el error de propagación durante los cálculos de ganancia de los árboles.
  • Memoria Contigua (C-Contiguous): El dataset se organiza en bloques de memoria físicamente adyacentes. Esto maximiza el Cache Locality, permitiendo que la CPU procese los datos sin saltos de memoria costosos.
ARQUITECTURA: Optimización para instrucciones SIMD (Single Instruction, Multiple Data).

Síntesis para la Defensa de Tesis

Pregunta Clave: ¿Cómo garantiza que sus datos son aptos para XGBoost?

Respuesta Maestra: Mediante un pipeline de tres capas: 1) Integridad: SQL con tipos exactos y joins estables. 2) Continuidad: Agregación temporal con imputación de ceros para evitar sesgos. 3) Inteligencia: Descomposición en features autorregresivas (Lags) y cíclicas que maximizan la ganancia de información en cada split del árbol.
